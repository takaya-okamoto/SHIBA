# データはどう保存されるか — 具体例で見る

> 1本の会話を例に、「LINEの会話 → Markdown(正本)→ TiDB(検索索引)」がどう保存されるかを追います。
> **大前提**: 記憶の正本は **Markdown + git**。TiDB はそれを速く検索するための**派生索引**(消しても `reindex` で復元可能)。

---

## 全体のディレクトリ

```
data/
├── memory/                       ← 🟢 記憶の正本(Markdown + git。GitHubへpushされる)
│   ├── MEMORY.md                 ← 常駐の小さな記憶(永続的な好み・人物)
│   ├── profile.md                ← オーナーのプロフィール
│   └── memory/
│       ├── 2026-06-13.md         ← 日次ノート(その日の会話ログ + Facts)
│       └── 2026-06-18.md
└── state/                        ← ⚪ ローカル状態(gitには入れない)
    └── sessions/
        └── U1234.../2026-06-13T2104.jsonl   ← 会話のトランスクリプト

TiDB(クラウド)                     ← 🔵 派生索引 + 🟡 運用状態(下記)
```

---

## STEP 1. 会話する(LINE)

```
あなた  「来週月曜にA社と打ち合わせ。担当は田中さん。」
SHIBA  「了解。6/16(月)のA社・田中さんとの打ち合わせ、覚えておきます。」
あなた  「あとコーヒーはブラック派です」
SHIBA  「ブラックコーヒー、覚えました ☕」
```

この時点では即座の応答だけ。記憶への書き込みは**セッションが閉じたとき**(無応答4時間 / 毎朝4時 / 50ターン到達)にまとめて行われます。

---

## STEP 2. 会話のトランスクリプト(JSONL・ローカル)

`data/state/sessions/U1234.../2026-06-13T2104.jsonl`(1行=1ターン):

```jsonl
{"v":1,"turn_id":"01J...","ts":"2026-06-13T21:04:05+09:00","role":"user","text":"来週月曜にA社と打ち合わせ。担当は田中さん。"}
{"v":1,"turn_id":"01J...","ts":"2026-06-13T21:04:07+09:00","role":"assistant","text":"了解。6/16(月)のA社・田中さんとの打ち合わせ、覚えておきます。"}
{"v":1,"turn_id":"01J...","ts":"2026-06-13T21:05:30+09:00","role":"user","text":"あとコーヒーはブラック派です"}
```

これは「何を話したか」の生ログ。記憶(下記)とは別物です。

---

## STEP 3. セッションが閉じる → 抽出 → Markdownに書く(正本)

背景処理が会話から**事実を抽出**し、日次ノートに書きます。

### `data/memory/memory/2026-06-13.md`

```markdown
---
date: 2026-06-13
---

## ログ

- A社との打ち合わせが来週月曜(2026-06-16)に決まった。担当は田中さん。
- コーヒーはブラックが好き。

## Facts

<!--- shiba:facts v1 -->
| # | claim                                       | kind       | confidence | valid_from | context |
| 1 | A社との打ち合わせは2026-06-16、担当は田中    | event      | 0.95       | 2026-06-16 | line    |
| 2 | コーヒーはブラックが好き                     | preference | 0.90       |            | line    |
<!--- shiba:facts:end -->
```

ポイント:
- **「ログ」= 自由文**(後で意味検索で広く拾う)。**「Facts」= 構造化された原子的事実**(正確な想起・矛盾解決用)。
- 相対表現「来週月曜」は **絶対日付 `2026-06-16` に解決**して保存(後で「いつ?」に答えられる)。
- これは**人間が読めるただのMarkdown**。エディタで直接直せるし、gitで履歴が残ります。

---

## STEP 4. TiDBに索引化される(派生・検索用)

上記Markdownから、検索のためのコピーがTiDBに入ります。**これは消してもMarkdownから作り直せます。**

### `chunks` テーブル(自由文の意味検索用)

| id | file_path | content | embedding | content_hash | noted_on |
|----|-----------|---------|-----------|--------------|----------|
| 101 | memory/2026-06-13.md | A社との打ち合わせが来週月曜(2026-06-16)に決まった。担当は田中さん。 | `[0.012, -0.045, ...]` | `a1b2c3…` | 2026-06-13 |
| 102 | memory/2026-06-13.md | コーヒーはブラックが好き。 | `[0.071, 0.003, ...]` | `d4e5f6…` | 2026-06-13 |

### `facts` テーブル(構造化された事実)

| id | claim | kind | confidence | valid_from | recorded_at | state | superseded_by | source_path | row_num |
|----|-------|------|-----------|-----------|-------------|-------|---------------|-------------|---------|
| 1 | A社との打ち合わせは2026-06-16、担当は田中 | event | 0.95 | 2026-06-16 | 2026-06-13 | active | NULL | memory/2026-06-13.md | 1 |
| 2 | コーヒーはブラックが好き | preference | 0.90 | NULL | 2026-06-13 | active | NULL | memory/2026-06-13.md | 2 |

- `embedding` と全文索引(`content`/`claim`)で、**ベクトル + 全文のハイブリッド検索**ができます。
- `source_path` + `row_num`(facts)・`file_path` + `content_hash`(chunks)で、**TiDBの行 ⇄ Markdownの該当箇所**を往復できる → だから再構築できる。

---

## STEP 4.5. 繋がりも索引化される(entity)

「A社」「田中」「打ち合わせ」のような**登場人物・モノ**は entity として正規化され、**どの fact がどの entity に触れたか**が記録されます。これが「繋がり」の正体です(辿る前に write 時に作っておく → 検索が速い)。

### `entities` テーブル(登場するもの)

| id | slug | name | kind | aliases |
|----|------|------|------|---------|
| 11 | tanaka | 田中 | person | ["田中さん","Tanaka"] |
| 12 | a-corp | A社 | org | [] |
| 13 | mtg-a | A社打ち合わせ | event | [] |
| 14 | coffee | コーヒー | topic | [] |

### `fact_entities` テーブル(fact ↔ entity の多対多)

| fact_id | entity_id |
|---------|-----------|
| 1 | 11(田中) |
| 1 | 12(A社) |
| 1 | 13(打ち合わせ) |
| 2 | 14(コーヒー) |

→ **fact #1 が「田中・A社・打ち合わせ」を1つに繋いでいる**(fact = ハイパーエッジ)。グラフDBもBFSも使わず、「同じ fact に出てくるか」で繋がりが決まります。

これで:
- **「田中さんに関すること全部」** = `fact_entities WHERE entity_id=11` の **1クエリ**。
- **「田中さんと繋がるもの」**(1ホップ)= 田中と同じ fact に出る entity(A社・打ち合わせ)を **1クエリ**で。

## STEP 5. 後日、矛盾が起きる(非破壊で更新)

```
(6/18)
あなた  「やっぱりA社じゃなくてB社と打ち合わせになった」
```

抽出時の **reconcile** が「fact #1 と矛盾する新情報」と判断 → **古い記憶を消さず、状態を変えて新しい記憶を足す**(非破壊UPDATE)。

### Markdown側: 古いノートの行に取り消し線、新しいノートに新事実

`memory/2026-06-13.md`(更新後):
```markdown
| 1 | ~~A社との打ち合わせは2026-06-16、担当は田中~~ | event | 0.95 | 2026-06-16 | superseded by 2026-06-18#1 |
| 2 | コーヒーはブラックが好き                       | preference | 0.90 |     | line |
```

`memory/2026-06-18.md`(新規):
```markdown
| 1 | 打ち合わせはB社に変更(2026-06-16) | event | 0.95 | 2026-06-16 | line |
```

### TiDB側: `facts` の状態が変わり、繋がりも新しい事実に張り替わる

| id | claim | state | superseded_by |
|----|-------|-------|---------------|
| 1 | A社との打ち合わせは… | **superseded** | **7** |
| 2 | コーヒーはブラックが好き | active | NULL |
| 7 | 打ち合わせはB社に変更(2026-06-16) | active | NULL |

新しい fact #7 が言及する entity(B社・田中・打ち合わせ)が `fact_entities` に追加され、`entities` には B社(id=15)が増えます:

| fact_id | entity_id |
|---------|-----------|
| 7 | 15(B社) |
| 7 | 11(田中) |
| 7 | 13(打ち合わせ) |

→ 以降「打ち合わせに関すること」を引くと、**active の #7(B社)** が返り、#1(A社)は履歴として残るだけ。**繋がりは常に「現在の真実」を指す**ので、「前はA社だったよね?」にも答えられるし、誤更新からの復旧もできます。

## STEP 6. 大事な好みは MEMORY.md に昇格する

「コーヒーはブラック」のように**何度も思い出される永続的な好み**は、夜間の整理(dreaming)で常駐記憶へ昇格します。

### `data/memory/MEMORY.md`(常に応答プロンプトに入る小さな記憶)

```markdown
# Long-Term Memory

## 好み
- コーヒーはブラック派 ☕
```

日次ノート(その日の細かい話)と、MEMORY.md(ずっと効く要点)の二段構えです。

---

## STEP 7. 思い出す(検索 → 応答)

```
(6/22)
あなた  「打ち合わせいつだっけ?」
```

1. クエリでTiDBをハイブリッド検索 → `facts #7`(active, B社, 2026-06-16)がヒット。
2. その記憶を **`<relevant-memories>` ブロックとして、その場限り**で応答プロンプトに注入(会話履歴には残さない)。
3. SHIBA:「**B社との打ち合わせは6/16(月)** です。担当は…(略)」

### 繋がりを使った想起の例

```
あなた  「田中さんとの予定ある?」
```

1. クエリから entity「田中」を解決(id=11)。
2. **entity-route**: `fact_entities WHERE entity_id=11` → 田中に繋がる active な fact(#7 = B社打ち合わせ 6/16)。表層に「打ち合わせ」という語が無くても、**繋がりで辿り着ける**。
3. SHIBA:「田中さんとは **6/16(月)にB社で打ち合わせ** の予定です。」

→ 「Aと言えばB」「Xに関すること全部」が、グラフを辿らず **1〜2クエリ**で返る。これがSHIBAの繋がり管理(write時に解決して索引化、read時は辿らない)です。

---

## (おまけ)秘密情報はどう保存される?

```
あなた  「APIキーは sk-abc123xyz... を使ってる」
```

SHIBAは**値そのものを記憶しません**。「設定されている事実」だけを残します:

`memory/...md`:
```markdown
| 3 | APIキーが設定済み(2026-06-22時点) | fact | 0.9 | | line |
```

→ 記憶・索引・ログのどこにも `sk-abc123xyz...` は保存されません(入力時にスクラブ)。

---

## まとめ

| 何が | どこに | 消えたら |
|------|--------|----------|
| 🟢 記憶の言葉(正本) | `data/memory/*.md`(git)+ GitHub | GitHubから復元 |
| 🔵 検索索引(chunks / facts) | TiDB | `reindex --all` でMarkdownから再構築 |
| 🟡 運用状態(recall統計・push予算など `st_*`) | TiDB + 週次スナップショット | 消失許容(再蓄積) |
| ⚪ 会話の生ログ | `data/state/sessions/*.jsonl` | 過去会話検索のみ失う |
| 🔴 秘密(APIキー等) | `.env`(0600) | 記憶にもログにも入れない |

**TiDBが万一全部消えても、`data/memory/` のMarkdownさえ無事なら記憶は完全に戻ります。** これがSHIBAの「Markdown=正本、DB=使い捨ての索引」という設計です。
